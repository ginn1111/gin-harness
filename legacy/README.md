# Hermes Profile Builder

A portable scaffold for creating and verifying the four **Hermes Solo Delivery** profiles:

| Profile | Role | Description |
|---------|------|-------------|
| `ginb` | Builder | Builds approved deliveries from the authoritative brief |
| `ginr` | Reviewer | Reviews completed deliveries against the brief |
| `gins` | Shipper | Final verification and delivery readiness |
| `gino` | Orchestrator | Decomposes L/XL work into M-or-smaller deliveries |

The source specification is preserved in [`docs/profile-creator-prompt.md`](docs/profile-creator-prompt.md).

## Quick start (new machine)

```bash
# 1. Check prerequisites
make doctor

# 2. Full bootstrap: install deps → create profiles → clone shared skills → register → verify
make setup
```

This is safe to run on a machine with existing profiles — `apply-new` skips targets that already exist.

## Step-by-step

```bash
# Check what's installed and what's missing
make doctor

# Install Python deps (PyYAML)
make doctor-deps

# Preview profile creation without touching anything
make dry-run-new

# Create the four delivery profiles (skips existing ones)
make apply-new

# Preview shared-skills cloning
make dry-run-shared

# Clone Matt Pocock skills repo + register in all profiles
make apply-shared

# Full verification pass
make verify
```

## Portable vs machine-specific

**In repo** — SOUL templates, config schema, scripts, tool proposals:
```
profiles/*.SOUL.md
config/profiles.yaml
toolsets/*.yaml
scripts/
templates/
```

**Machine-specific (not in repo)** — generated from templates on setup:

| Artifact | Source |
|----------|--------|
| `~/.hermes/profiles/{ginb,gino,ginr,gins}/config.yaml` | Generated by `hermes profile create` |
| `~/.hermes/profiles/{ginb,gino,ginr,gins}/.env` | Machine credentials (kept out of repo) |
| `.hermes/reports/*` | Inspection output |
| `.hermes/plans/*` | Execution plans |

`.gitignore` already covers these — no accidental commits.

## Safety model

- Commands are dry-run-first (`make dry-run-new`, `make dry-run-shared`).
- Existing target profiles are never overwritten.
- New targets use `--no-skills` (bundled skills disabled).
- Old profile skills, toolsets, memory, sessions, credentials, and environment configuration are never cloned.
- Role-specific skills start empty.
- Toolset files are proposals with `pending_human_definition` status.
- Shared Matt Pocock skills are cloned once and registered in all profiles.

## Shared skill updates

The shared repository is made read-only after installation. To update:

```bash
chmod -R u+w "$HOME/.hermes/shared-skills/mattpocock-skills"
git -C "$HOME/.hermes/shared-skills/mattpocock-skills" pull --ff-only
chmod -R a-w "$HOME/.hermes/shared-skills/mattpocock-skills"
```

Review upstream changes before updating.

## Legacy migration

Read-only discovery:

```bash
make discover
```

Then complete `templates/migration-mapping.md` before applying. Full sequence:

1. Discover old profiles.
2. Approve old-to-new mapping.
3. Resolve target conflicts.
4. Create clean targets (`make apply-new`).
5. Selectively migrate personality only.
6. Review migrated personality.
7. Register shared skills (`make apply-shared`).
8. Add explicitly approved role-specific skills.
9. Approve and configure least-privilege toolsets.
10. Verify targets (`make verify`).

## Make targets

| Target | Purpose |
|--------|---------|
| `doctor` | Check prerequisites (CLI, deps, profiles, skills) |
| `doctor-deps` | Install Python dependencies |
| `setup` | Full bootstrap: deps → profiles → skills → verify |
| `dry-run-new` | Preview profile creation |
| `apply-new` | Create delivery profiles (safe, skips existing) |
| `dry-run-shared` | Preview shared-skill cloning |
| `apply-shared` | Clone + register shared skills |
| `verify` | Full delivery profile verification |
| `discover` | Read-only existing-profile inventory |
| `lint` | Shell + Python syntax checks |
